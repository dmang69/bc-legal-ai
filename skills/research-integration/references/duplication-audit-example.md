## Audit Date: [YYYY-MM-DD]

## Findings

### Finding 1 — Definitional Overlap

**Concept:** Ontological categories vs observational groupings

**Where duplicated:**
- energy-regimes-thesis: Section on "regime as ontological category"
- quantum-consciousness-thesis: Chapter defining "ontological category" for
  Orch-OR discussion
- reality-desync: Substrate ontology chapter

**Type:** Same idea, three independent introductions

**Cost of leaving as-is:**
- Each work reintroduces the concept, adding pages that could be citations
- Inconsistent phrasing across works
- Reader who reads two of them sees redundancy

**Fix:**
- Designate energy-regimes-thesis as canonical treatment
- quantum-consciousness and reality-desync: cite energy-regimes for the
  framework, briefly note the specialization to their domain, do not
  redevelop
- Estimated pages saved across portfolio: 12-18

**Priority:** Medium (works are already published/drafted; fix on next revision)

---

### Finding 2 — Method Overlap

**Method:** Signal-signature-based measurement

**Where duplicated:**
- echo-calculus-thesis: Introduces Σ quintuple
- echo-calculus-sif: Redefines with additional formal structure
- ai-native-internet: Uses the concept for benchmarks

**Type:** Progressive development — NOT true duplication

**Analysis:**
This isn't duplication, this is refinement. The concept genuinely develops
across the three works. echo-calculus introduces, sif formalizes,
ai-native-internet applies.

**Fix:**
- No merge needed
- ADD cross-citations: ai-native-internet should cite sif for the formal
  properties it relies on
- ADD reference in echo-calculus to the later refinement

**Priority:** Low (works are complementary, just need explicit citation)

---

### Finding 3 — Case-Study Overlap

**Case:** [Specific benchmark or example used in multiple works]

**Where duplicated:**
- ai-native-internet: Uses [benchmark] for KPI validation
- echo-calculus-sif: Uses [benchmark] for measurement validation

**Type:** Same evidence, different claims

**Analysis:**
Using the same benchmark for two claims is fine IF the claims are
independent. If they're related, one work should cite the other rather than
re-running the analysis.

**Fix:**
- Check if the benchmark analyses in the two works are identical or if
  each does something different with the same data
- If identical: designate one as canonical, other cites
- If different: cross-reference so the reader can see the full picture

**Priority:** Low

## Non-Findings (Cleared)

Ideas that appear in multiple works but do NOT constitute duplication:

- Capability tokens (intentkernel-os) and Legal standing (BC JR):
  Related by analogy only. Different formal treatment appropriate.

- Physics ontology (energy-regimes) and Software ontology (intentkernel):
  Related concept applied to different substrates. Each treatment is
  domain-appropriate. Only reconcile the philosophical framework, not the
  substantive content.

## Audit Discipline

Run this audit:
- Every ~6 months
- Before submitting any new work (to check for new duplications)
- After finishing any new work (to see what the new work should cite)

The audit is not a criticism of past work. It's a maintenance pass on a
research portfolio — the same discipline applied to a codebase.
